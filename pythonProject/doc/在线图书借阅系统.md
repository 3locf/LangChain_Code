### 3. 介绍一下这个项目

- 回答一：(精简版)

  这个项目是一个功能完善、操作便捷的在线图书借阅系统，主要面向学校图书馆或小型社区图书馆使用。系统分为管理员端和用户端两个部分：管理员可以管理图书信息、用户信息、借阅记录，处理图书入库和出库；用户可以查询图书、在线借阅、预约图书、查看自己的借阅历史和逾期情况。

系统实现了完整的图书借阅流程，包括图书查询、借阅申请、图书归还、逾期罚款计算等核心功能。针对高并发借阅场景，我做了库存锁和异步处理优化；同时使用定时任务实现了逾期提醒和自动罚款功能。

- 回答二：(完整版 [根据面试官状态选择回答])

1. 项目背景：大学期间，我发现学校图书馆的借阅系统存在操作繁琐、响应缓慢、信息更新不及时等问题，给师生带来了很多不便。同时我也希望通过一个完整的项目来巩固自己的 Java 后端开发技能，于是决定开发一个更高效、更易用的在线图书借阅系统。
2. 项目目标：开发一个功能全面、性能稳定、安全可靠的在线图书借阅系统，实现图书管理、用户管理、借阅管理、预约管理等核心功能，提高图书馆的工作效率和用户体验。
3. 个人角色：这是一个前后端分离的 Web 项目，我负责整个项目的需求分析、架构设计、数据库设计以及后端全部功能的开发。前端页面由我的同学负责开发，我同时也负责前后端接口的联调和项目的整体进度把控。
4. 项目设计：项目采用了经典的分层架构和模块化设计。
   - 分层：展示层、控制层、服务层、数据访问层
   - 分模块：用户模块、图书模块、借阅模块、预约模块、系统管理模块
5. 项目功能介绍:
   - 用户模块：实现了用户注册、登录、个人信息管理、权限控制等功能，分为普通用户和管理员两种角色。
   - 图书模块：支持图书信息的增删改查、图书分类管理、图书库存管理、图书导入导出等功能。
   - 借阅模块：实现了完整的借阅流程，包括图书借阅、图书归还、借阅记录查询、逾期罚款计算等。
   - 预约模块：用户可以预约已被借出的图书，当图书归还时系统会自动通知预约用户。
   - 系统管理模块：包括系统参数配置、日志管理、数据备份等功能。
6. 技术栈：在项目中，我使用了:
   - Spring Boot3 + JDK 17
   - 数据库: MySQL + MyBatis
   - 缓存: Redis
   - 消息队列: RabbitMQ
   - 定时任务: Quartz
   - 日志: SLF4J + logback
   - 身份验证: JWT
7. 挑战和解决方案:
   - 图书库存的并发安全问题：当多个用户同时借阅同一本图书时，可能会出现超借的情况。我使用 Redis 分布式锁解决了这个问题，保证了库存操作的原子性。
   - 系统性能优化：图书查询是系统中最频繁的操作，我将热门图书信息和分类信息存入 Redis 缓存，大大提高了查询速度。同时将邮件通知、短信提醒等耗时操作改为异步处理，提升了系统的响应速度。
   - 定时任务的实现：系统需要每天自动计算逾期罚款并发送提醒通知，我使用 Quartz 定时任务框架实现了这个功能，并支持任务的动态配置和管理。
8. 学习和成长:
   - 掌握了完整的 Web 项目开发流程，从需求分析、设计、开发到测试部署的全过程。
   - 深入理解了并发编程、分布式锁、消息队列、缓存等后端核心技术的原理和应用。
   - 学会了如何进行数据库设计和优化，如何保证系统的性能和稳定性。

### 5. 你在这个项目中使用了哪些技术？为什么选择 (决策) 这些技术？

- 关联问题:
  - 为什么缓存选择 Redis?
  - 为什么消息队列选择 RabbitMQ?
  - 为什么定时任务选择 Quartz?
- 数据库: MySQL + MyBatis
  - MySQL 是一个开源免费的关系型数据库，性能稳定，社区活跃，是中小型项目的首选。
  - MyBatis 是一个轻量级的 ORM 框架，它允许开发者直接编写 SQL 语句，灵活性高，易于优化，非常适合复杂的业务查询场景。
- 缓存: Redis
  - 图书查询是系统中最频繁的操作，尤其是热门图书和分类信息，访问量非常大。将这些数据存入 Redis 缓存可以显著提高查询速度，减轻数据库的压力。
  - Redis 的其他特点:
    - 高性能：基于内存的存储系统，读写速度极快。
    - 丰富的数据结构：支持字符串、哈希、列表、集合、有序集合等多种数据结构，能够满足不同的缓存需求。
    - 持久化支持：提供 RDB 和 AOF 两种持久化方式，保证数据的安全性。
    - 分布式锁：可以方便地实现分布式锁，解决并发安全问题。
    - 高并发处理能力：能够处理大量的并发请求，非常适合高流量的 Web 应用。
- 消息队列: RabbitMQ
  - 系统中有很多耗时的操作，比如邮件通知、短信提醒、数据统计等，如果同步执行会严重影响系统的响应速度。使用 RabbitMQ 可以将这些操作异步化，提高系统的吞吐量和用户体验。
  - RabbitMQ 的其他特点:
    - 异步处理能力：将耗时任务异步处理，提高系统响应速度。
    - 解耦和灵活性：生产者和消费者之间解耦，提高系统的可维护性和扩展性。
    - 可靠性和持久性：支持消息持久化和可靠传输，确保消息不丢失。
    - 消息确认和死信队列：提供完善的消息确认机制和死信队列，保证消息的可靠消费。
    - 社区支持和成熟性：开源项目，社区活跃，稳定性高。
- 定时任务: Quartz
  - 系统需要每天自动计算逾期罚款、发送逾期提醒、清理过期数据等，这些都需要定时任务来实现。Quartz 是一个功能强大的定时任务框架，支持复杂的任务调度、任务持久化、集群部署等功能，能够满足系统的需求。
- 身份验证: JWT
  - 系统采用前后端分离架构，使用 JWT 进行身份验证非常方便。JWT 不需要在服务器端保存会话信息，减轻了服务器的压力，同时也便于系统的横向扩展。
  - JWT 是行业标准，安全性高，支持跨域访问，非常适合前后端分离的项目。
- 隐私数据加密: sha256 + aes
  - 用户密码使用 SHA-256 加盐哈希加密，保证密码的安全性，即使数据库泄露也无法还原用户的原始密码。
  - 用户的手机号、邮箱等敏感信息使用 AES 对称加密存储，在需要使用时再解密，保护用户的隐私。

### 7. 在项目中，你是如何平衡技术实现和项目时间线的？这个项目做了多长时间？

这个项目我总共花了 3 个月的时间完成，从需求分析、原型设计、数据库设计，到前后端开发、测试、部署上线。

关于技术实现和项目时间线的平衡，我主要采取了以下措施:

1. 明确项目范围和目标：在项目开始之前，我就明确了项目的核心功能和非核心功能，优先保证核心功能的实现，非核心功能可以在后续版本中迭代开发。
2. 制定详细的项目计划：我将整个项目划分为多个阶段，每个阶段都有明确的交付物和截止时间。比如第一周完成需求分析和原型设计，第二周完成数据库设计和架构搭建，第三到第六周完成核心功能开发，第七到第八周完成测试和 bug 修复，第九周完成部署上线。
3. 功能优先级排序：我将功能按照重要性和紧急程度进行了排序，优先实现用户最常用的功能，比如图书查询、借阅、归还等。对于一些锦上添花的功能，比如图书推荐、评论等，放在了后续版本中。
4. 合理选择技术：在技术选型上，我优先选择自己熟悉的、成熟稳定的技术，避免使用过于复杂或不成熟的技术，以免影响项目进度。
5. 及时沟通和协作：我和前端同学保持密切的沟通，每周至少进行一次进度同步，及时解决开发中遇到的问题，确保前后端开发进度一致。
6. 保证代码质量：在开发过程中，我注重代码的规范性和可维护性，编写了必要的单元测试，减少了后期 bug 修复的时间。

### 8. 你在项目中遇到了哪些挑战？你是如何识别和解决这些问题的？

非技术方面:

- 需求不明确：由于没有专业的产品经理，很多需求需要我自己去调研和梳理。我通过采访学校图书馆的老师和同学，了解他们的实际需求和使用习惯，然后整理成详细的需求文档。同时在开发过程中，我也会不断和用户沟通，及时调整需求。
- 项目进度管理：这是我第一次独立负责一个完整的项目，在进度管理方面缺乏经验。我通过制定详细的项目计划，将大任务分解成小任务，每天记录自己的工作进度，及时发现和解决进度滞后的问题。

技术方面:

- 图书库存的并发安全问题：当多个用户同时借阅同一本图书时，可能会出现超借的情况。我最初使用 synchronized 关键字来解决这个问题，但在分布式环境下无法生效。后来我学习了 Redis 分布式锁的原理，使用 Redisson 框架实现了分布式锁，完美解决了这个问题。
- 系统性能优化：随着图书数量和用户数量的增加，图书查询接口的响应速度越来越慢。我通过分析慢查询日志，发现是因为没有给常用的查询字段建立索引。我给图书表的书名、作者、分类等字段建立了索引，同时将热门图书信息存入 Redis 缓存，查询速度提升了 10 倍以上。
- 定时任务的可靠性问题：系统需要每天凌晨自动计算逾期罚款，如果服务器在这个时间点宕机，就会导致任务执行失败。我使用 Quartz 的持久化功能，将任务信息存入数据库，即使服务器宕机，重启后也能继续执行未完成的任务。同时我还添加了任务执行日志和告警功能，方便问题排查。

### 9. 你是如何测试和部署项目的？

测试:

1. 单元测试：我使用 JUnit 框架编写了单元测试，覆盖了所有核心业务逻辑，确保每个方法的正确性。
2. 接口测试：使用 Postman 对所有后端接口进行了测试，验证接口的返回结果是否符合预期，以及异常情况的处理是否正确。
3. 集成测试：在前后端联调阶段，我和前端同学一起进行了集成测试，测试各个模块之间的交互是否正常。
4. 功能测试：邀请了一些同学作为测试用户，对系统的所有功能进行了全面的测试，发现并修复了很多 bug。
5. 性能测试：使用 JMeter 对系统进行了压力测试，测试系统在高并发情况下的性能表现，并根据测试结果进行了优化。
6. 安全测试：测试了系统的身份验证、权限控制、SQL 注入防护等安全功能，确保系统的安全性。

部署:

1. 环境准备：在阿里云服务器上安装了 JDK17、MySQL8.0、Redis6.0、RabbitMQ3.9 等软件，并进行了相应的配置。
2. 项目构建：使用 Maven 将项目打包成 jar 包。
3. 数据库部署：在服务器上创建了数据库，并执行了 SQL 脚本初始化数据库结构和基础数据。
4. 项目部署：将 jar 包上传到服务器，使用 nohup 命令后台启动项目。
5. 配置反向代理：使用 Nginx 作为反向代理服务器，将请求转发到 Spring Boot 应用，同时配置了 HTTPS, 提高系统的安全性。
6. 日志管理：配置了 logback 将日志输出到文件，并使用 logrotate 进行日志切割和归档，方便问题排查。

### 10. 你是如何优化项目性能的？你有哪些性能调优的经验？

- 数据库优化:
  - 给常用的查询字段建立索引，避免全表扫描。
  - 优化 SQL 语句，避免使用 SELECT *, 减少不必要的数据传输。
  - 使用分页查询，避免一次性查询大量数据。
  - 对大表进行分表处理，提高查询效率。
- 缓存优化:
  - 将热门图书信息、分类信息、系统参数等频繁访问的数据存入 Redis 缓存，减少数据库的访问次数。
  - 合理设置缓存的过期时间，避免缓存数据过期导致的缓存雪崩问题。
  - 使用布隆过滤器解决缓存穿透问题。
- 异步处理:
  - 将邮件通知、短信提醒、数据统计等耗时操作改为异步处理，使用 RabbitMQ 消息队列实现，提高系统的响应速度。
  - 使用多线程处理批量操作，比如批量导入图书、批量发送通知等。
- 代码优化:
  - 避免在循环中操作数据库，尽量使用批量操作。
  - 使用连接池管理数据库连接和 Redis 连接，避免频繁创建和销毁连接。
  - 合理使用设计模式，提高代码的执行效率和可维护性。
- 服务器优化:

### 11. 你是如何确保项目安全性的？你使用了哪些安全措施？

  - 限制用户的登录尝试次数，防止暴力破解。
  - 记录用户的操作日志，方便安全审计和问题排查。

## 专属问题

### 1. 图书借阅流程是如何实现的？如何避免超借和逾期？

图书借阅流程如下:

1. 用户登录系统，查询想要借阅的图书。
2. 用户点击借阅按钮，系统首先检查用户是否有未还的逾期图书，以及用户的借阅数量是否已达上限。
3. 系统检查图书的库存是否充足，如果充足则扣减库存，生成借阅记录，并将图书状态改为已借出。
4. 系统返回借阅成功的信息，并显示图书的应还日期。

为了避免超借:

- 系统对每个用户的最大借阅数量进行了限制，学生最多可以借阅 10 本，教师最多可以借阅 20 本。
- 使用 Redis 分布式锁保证库存扣减操作的原子性，防止多个用户同时借阅同一本图书导致超借。
- 在数据库层面也添加了库存约束，确保库存不会出现负数。

为了避免逾期:

- 用户借阅图书时，系统会明确显示应还日期，并在用户的个人中心显示所有未还图书的到期时间。
- 系统会在图书到期前 3 天和到期当天，通过邮件和短信向用户发送提醒通知。
- 对于逾期未还的图书，系统会每天自动计算逾期罚款，并限制用户继续借阅新的图书，直到归还所有逾期图书并缴纳罚款。

### 2. 如何处理图书归还时的逾期罚款计算？

我使用 Quartz 定时任务框架实现了逾期罚款的自动计算功能:

1. 系统每天凌晨 1 点会执行一个定时任务，查询所有未还的借阅记录。
2. 对于每条借阅记录，系统计算当前日期与应还日期的差值，如果差值大于 0, 则表示图书已逾期。
3. 系统根据逾期天数和图书类型计算逾期罚款，学生的罚款标准是 0.1 元 / 天，教师的罚款标准是 0.05 元 / 天。
4. 系统将罚款信息更新到用户的账户中，并生成罚款记录。
5. 系统向逾期用户发送邮件和短信提醒，通知他们尽快归还图书并缴纳罚款。

当用户归还图书时:

1. 系统首先检查用户是否有逾期罚款，如果有则需要先缴纳罚款才能归还图书。
2. 系统增加图书的库存，将图书状态改为可借阅。
3. 系统更新借阅记录的归还时间和状态，并生成归还记录。

### 3. 为什么要使用 Redis 缓存图书信息和借阅记录？

使用 Redis 缓存图书信息和借阅记录主要有以下几个原因:

1. 提高查询速度：图书查询是系统中最频繁的操作，尤其是热门图书，访问量非常大。Redis 是基于内存的存储系统，读写速度极快，可以显著提高查询速度，减轻数据库的压力。
2. 减轻数据库负担：如果每次查询都直接访问数据库，会给数据库带来很大的压力，甚至导致数据库崩溃。使用 Redis 缓存可以将大部分查询请求拦截在缓存层，只有缓存未命中时才会访问数据库。
3. 支持高并发: Redis 具有很高的并发处理能力，能够轻松应对大量的并发查询请求，非常适合高流量的 Web 应用。
4. 实现分布式锁：我使用 Redis 实现了分布式锁，解决了图书借阅时的库存并发安全问题。
5. 缓存借阅记录：用户经常需要查询自己的借阅历史，将用户的最近借阅记录存入 Redis 缓存，可以提高查询速度，提升用户体验。

我采用了 "缓存 + 数据库" 的读写模式:

- 读操作：先从缓存中读取，如果缓存中没有，再从数据库中读取，并将结果存入缓存。
- 写操作：先更新数据库，然后再更新缓存或者删除缓存。

### 4. 如何保证图书库存的并发安全性？

当多个用户同时借阅同一本图书时，可能会出现超借的情况，这就是典型的并发安全问题。我使用 Redis 分布式锁解决了这个问题，具体实现如下:

1. 当用户点击借阅按钮时，系统首先尝试获取 Redis 分布式锁，锁的 key 是 "book:lock:{bookId}",value 是一个唯一的请求 ID, 锁的过期时间设置为 30 秒。
2. 如果获取锁成功，则执行库存扣减和借阅记录生成的操作。
3. 操作完成后，释放分布式锁。
4. 如果获取锁失败，则表示有其他用户正在借阅这本图书，系统返回 "图书正在被借阅，请稍后再试" 的提示。

为了防止锁被永久占用，我给锁设置了过期时间，即使服务宕机，锁也会在 30 秒后自动释放。同时我还使用了 Lua 脚本保证了锁的释放操作的原子性，避免误删其他线程的锁。

另外，我还在数据库层面添加了库存约束，使用乐观锁进行二次校验，确保库存不会出现负数:

```
UPDATE book SET stock = stock - 1 WHERE id = #{bookId} AND stock > 0
```

### 5. 如何实现图书的预约和通知功能？

图书预约功能的实现流程:

1. 用户查询图书时，如果图书已被借出，系统会显示 "预约" 按钮。
2. 用户点击预约按钮，系统检查用户是否已经预约过这本图书，以及用户的预约数量是否已达上限。
3. 如果检查通过，系统生成预约记录，并将预约状态设置为 "待通知"。
4. 系统返回预约成功的信息，并告知用户当图书归还时会收到通知。

通知功能的实现:

1. 当用户归还图书时，系统检查是否有用户预约了这本图书。
2. 如果有预约记录，系统将预约状态改为 "已通知", 并通过 RabbitMQ 发送一条通知消息。
3. 消息消费者接收到通知消息后，通过邮件和短信向预约用户发送通知，告知他们图书已归还，请在 3 天内前来借阅。
4. 如果预约用户在 3 天内没有借阅这本图书，系统会自动取消预约，并将图书分配给下一个预约用户。

我使用 RabbitMQ 实现了异步通知，这样即使邮件或短信服务出现问题，也不会影响图书归还的主流程。同时我还使用了死信队列，确保通知消息不会丢失，如果通知失败，会进行重试。
